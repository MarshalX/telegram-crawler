Users can create managed bots directly through the MTProto API, without interacting with @BotFather.

A managed bot has two related accounts:

  • The user account that creates and owns the bot.
  • The manager bot passed in bots.createBot.manager_id. The manager bot must be a user with both the bot and bot_can_manage_bots flags set; bot_can_manage_bots means that the bot can manage bots created by users through this flow.

Direct creation

Before invoking bots.createBot, call bots.checkUsername with the full target username. The username must end in bot; clients may append this suffix in the UI, but the string passed to the API must already contain it.

Then invoke bots.createBot:

  • name is the bot display name, 1-64 characters.
  • username is the checked username, including the bot suffix.
  • manager_id is the manager bot.
  • Set via_deeplink only if the creation prompt was opened from a managed bot deep link ».

The returned user is the created bot. Store it in the peer database » like any other received user.

If the current user already owns the maximum allowed number of bots, the server may return BOT_CREATE_LIMIT_EXCEEDED. The limit is exposed through the bots_create_limit_default » and bots_create_limit_premium » config keys, and as the double_limits__bots_create » Premium limit. Clients should offer to open https://t.me/BotFather?start=deletebot so the user can delete an owned bot.

Other errors handled by Telegram clients include:

  • MANAGER_PERMISSION_MISSING if manager_id is not allowed to manage newly created bots.
  • USERNAME_INVALID or USERNAME_OCCUPIED if the username cannot be used.
  • FLOOD_WAIT_%d if bot creation is temporarily rate-limited.

Request buttons

A manager bot may ask a user to create a bot by sending an inputKeyboardButtonRequestPeer button whose RequestPeerType is requestPeerTypeCreateBot. User clients receive it as keyboardButtonRequestPeer. Such buttons are available only in private chats.

When the user presses the button, show a creation prompt using:

  • suggested_name, if set, as the initial bot name.
  • suggested_username, if set, as the initial bot username. Clients may display the editable part without the trailing bot, but must pass the full username to bots.checkUsername and bots.createBot.

After bots.createBot succeeds, invoke messages.sendBotRequestedPeer:

  • peer is the manager bot.
  • msg_id is the message containing the keyboardButtonRequestPeer.
  • button_id is the button's button_id.
  • requested_peers contains the newly created bot.

The manager bot receives a messageActionRequestedPeer service message as described in the button flow ».

Deep links

Clients should parse the following links as managed bot creation links:

  • https://t.me/newbot/<manager_bot_username>/<suggested_bot_username>?name=<suggested_bot_name>
  • https://t.me/newbot/<manager_bot_username>?name=<suggested_bot_name>
  • tg://newbot?manager=<manager_bot_username>&username=<suggested_bot_username>&name=<suggested_bot_name>

When opening such a link:

  1. Resolve manager_bot_username.
  2. Check that the resolved peer is a bot and that user.bot_can_manage_bots is set.
  3. Show the same creation prompt used for request buttons ».
  4. If the user confirms, invoke bots.createBot with via_deeplink set.

If suggested_bot_username does not end in bot, TDLib appends bot to the parsed suggestion before showing the prompt.

Mini Apps

A manager bot can prepare a request button for a Mini App » by invoking bots.requestWebViewButton with a KeyboardButton containing requestPeerTypeCreateBot. The manager bot should pass the returned bots.requestedButton.webapp_req_id to the Mini App.

When the Mini App emits web_app_request_chat », the user client must invoke bots.getRequestedWebViewButton, passing the manager bot and the event's req_id as webapp_req_id.

If the returned button is a keyboardButtonRequestPeer with requestPeerTypeCreateBot, show the creation prompt. After creating the bot, invoke messages.sendBotRequestedPeer with:

  • peer set to the manager bot.
  • webapp_req_id set to the Mini App request ID.
  • button_id set to the returned button's button_id.
  • requested_peers containing the newly created bot.

Notify the Mini App with requested_chat_sent » after messages.sendBotRequestedPeer succeeds, or requested_chat_failed » if the user cancels or the request fails.

Manager bot updates

The manager bot receives updateManagedBot after a user creates a bot managed by it:

  • user_id is the user who created the managed bot.
  • bot_id is the managed bot.

The manager bot can then invoke bots.exportBotToken for bot_id to obtain the managed bot token. Set revoke to replace the current token.

Manager bots can use bots.getAccessSettings and bots.editAccessSettings to read and update bots.accessSettings. If restricted is set, only the owner and the users listed in add_users can access the managed bot.

The user's chat with the manager bot may contain a messageActionManagedBotCreated service message; its bot_id is the newly created managed bot.
