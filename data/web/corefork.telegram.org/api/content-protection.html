Content protection

Users and bots who want to keep their messages private, as well as Group and Channel owners who want to keep their content members-only can enable content protection, which prevents screenshots, copying, disables message forwards and limits the ability to save media from posts.

messages.toggleNoForwards#b2081a35 flags:# peer:InputPeer enabled:Bool request_msg_id:flags.0?int = Updates;

To enable or disable content protection in a basic group, supergroup or channel simply invoke messages.toggleNoForwards, passing true or false to enabled (ignore the request_msg_id flag, it's only used when protecting private chats »).

This method can only be invoked by the owner of a group/channel (so only by users, as bots cannot own groups/channels).

Groups and channels where content protection is enabled will have the chat.noforwards/channel.noforwards flag set.

Note that the message.noforwards flag will not be set for groups/channels with channel/group-wide protection enabled, as that flag is only usable by bots »: however, all messages received from protected groups/chats must still be treated as if the message.noforwards flag was set (i.e. forwards, downloads, copying, screenshots must be disabled).

Attempting to forward messages from a protected chat/channel will emit a CHAT_FORWARDS_RESTRICTED RPC error.

// All other message sending methods also have the `noforwards` flag

Bots can send standalone protected messages to any private chat, group or channel, regardless of the chat-wide protection status, by simply setting the noforwards flag of messages.sendMessage and all other message sending methods.

This will set the message.noforwards flag of the outgoing messages; for these messages, forwards, downloads, copying, screenshots must be disabled.

Attempting to forward protected messages will emit a CHAT_FORWARDS_RESTRICTED RPC error.

---functions---

messages.toggleNoForwards#b2081a35 flags:# peer:InputPeer enabled:Bool request_msg_id:flags.0?int = Updates;

To enable or disable content protection in a private chat simply invoke messages.toggleNoForwards, passing true or false to enabled.

This method can only be invoked by users, in a private chat with another user: private chats with bots and official Telegram support accounts cannot be protected.

A Telegram Premium subscription is required to protect private chats (pm_noforwards feature identifier »), groups and channels can be protected even without a Premium subscription.

Protecting/unprotecting a private chat will emit a messageActionNoForwardsToggle service message.

This service message will only be emitted if:

  • The protection state actually changed (i.e. re-protecting a protected chat or unprotecting an unprotected chat will not emit updates nor RPC errors) OR
  • A valid request_msg_id is provided, in which case a messageActionNoForwardsToggle will always be emitted even when not changing the protection state (in this case, both prev_value and new_value will be equal to the current protection state), in order to mark the messageActionNoForwardsRequest as read (see below).

Private chats where content protection was enabled by our side/the other side will have the userFull.noforwards_my_enabled/noforwards_peer_enabled flag set.
Only one of the two flags will be set, depending on who disabled forwards first; if the chat was protected by the other side first, re-protecting the private chat on our side won't change the values of the flags (and vice versa).

If the noforwards_peer_enabled flag is set (i.e. the chat was protected by the other peer, not by us), invoking messages.toggleNoForwards will not unprotect the chat: instead, a messageActionNoForwardsRequest service message will be emitted, indicating to the other peer that we requested to disable content protection in the chat.

The other peer can then choose to accept or refuse the messageActionNoForwardsRequest by invoking messages.toggleNoForwards, populating the enabled flag with true to refuse the request, false to accept the request and the request_msg_id flag with the ID of the messageActionNoForwardsRequest service message; this will have the following consequences:

  • A new messageActionNoForwardsToggle will be emitted replying to the messageActionNoForwardsRequest (i.e. reply_to will be populated with the ID of the messageActionNoForwardsRequest).

    As mentioned above, this service message will always be emitted, even if the protection state wasn't changed (i.e. when re-protecting a protected chat or unprotecting an unprotected chat by clicking on the accept/refuse buttons of multiple requests in the same chat), in order to mark the messageActionNoForwardsRequest as read.

    Note that i.e. accepting a request and then refusing the next will have the effect of disabling and then re-enabling content protection in the chat.

  • The mentioned messageActionNoForwardsRequest will be automatically expired, setting the messageActionNoForwardsRequest.expired flag on the server (i.e. the updated value will be available to clients fetching the message history/updates): attempting to reuse the same request_msg_id will emit a REQUEST_MSG_EXPIRED RPC error.

  • Note that no update will be emitted to explicitly set the messageActionNoForwardsRequest.expired flag, if the original messageActionNoForwardsRequest was already received by the client using updates: clients should automatically expire locally cached messageActionNoForwardsRequests as soon as a messageActionNoForwardsToggle replying to the request is received.

    Unrelated messageActionNoForwardsToggles not replying to any request, or replying to other requests must not expire unrelated pending requests.

    Pending requests also won't be expired if protection is disabled by the side that enabled it without passing a request_msg_id.

Pending requests will automatically expire no_forwards_request_expire_period » seconds after they're sent, unless they are manually accepted or rejected, first.

Note that the message.noforwards flag will not be set for private chats with protection enabled, as that flag is only usable by bots »: however, all messages received from protected private chats must still be treated as if the message.noforwards flag was set (i.e. forwards, downloads, copying, screenshots must be disabled).

Attempting to forward messages from a protected private chat will emit a CHAT_FORWARDS_RESTRICTED RPC error.
